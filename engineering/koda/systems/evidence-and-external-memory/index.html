Evidence and External Memory

A record discipline for preserving context, provenance and decisions across a long-running solo engineering programme. Memory is not treated as proof, and documentation is not treated as implementation evidence by itself.

CurrentLast reviewed: 15 July 2026Evidence classes: IMPLEMENTED · VALIDATED · HISTORICAL

Objective and components

The system uses project-state records, handovers, decision locks, append-only logs, completion reports, validation reports and evidence packages. It preserves both positive results and constraints such as supersession, contradiction, missing identity or an unapproved next step.

Memory

Context that allows work to resume after interruption.

Evidence

Material supporting a bounded claim, method and limitation.

Canon

A reviewed state or decision, not merely an available file.

Provenance

Lineage and references that make a result inspectable.

Evidence-state progression

Validation and recovery

Existing results show append-only logging, immutable artefact descriptors, recovery inspection and evidence contracts in bounded areas of the programme. The record also retains failures and NO-GO outcomes, allowing a recovery path to start from a known boundary rather than from assumed memory.

Current state and planned development

This system is current as a working discipline and partially implemented infrastructure. Remaining work includes durable relationship identities and clearer contradiction/supersession links where live runtime records need them. Public documentation excludes internal record locations and sensitive raw material.

Related agent governance relies on evidence for review; Forge Workshop provides a human-facing surface for evidence-return; Nano Factory uses provenance to bound research claims.
